================================================================================
1. CONTEXTE
================================================================================

Retour beta-testeur (Master 2, cours d'algorithmie) : ~45 s d'attente pour
10 questions, ecran percu comme "casse". Modele actuel : Llama 3.1 8B en local
via Ollama (CPU). Objectif : ramener la generation a un temps acceptable
(cible interne <= 15 s) SANS sacrifier la differenciation RGPD local-first
(aucune donnee hors serveur) ni la qualite pedagogique des QCM en francais.

Charge reelle reproduite (cf. backend/llm/services/) :
  - Prompt = SYSTEM_PROMPT + cours tronque a MAX_SOURCE_CHARS = 8000 caracteres.
  - Appel Ollama /api/generate : stream=false, temperature=0.4, format=json.
  - Sortie attendue : JSON strict, 10 questions x 4 options.


================================================================================
2. PROTOCOLE DE BENCHMARK (CA-J2-2)
================================================================================

  Machine de ref   : laptop 16 Go RAM, CPU 8 coeurs recent, sans GPU dedie
                     (= cible kit CA-7 "laptop 16 Go"). Pas d'acceleration Metal.
  Outil            : Ollama + scripts/benchmark_llm.py (charge identique a l'app)
  Cours de ref     : "Algorithmique - Complexite et tris" (~3,5k car.), identique
                     pour tous les modeles
  Runs             : 5 mesures + 1 chauffe (modele charge en RAM) par modele
  Parametres       : strictement ceux de l'app (section 1)
  Metriques        : latence end-to-end p50 (mediane) et p95 ; debit tok/s ;
                     taille disque/RAM ; qualite /5 ; taux de sorties valides.

  Sources des chiffres :
    - Latence du modele ACTUEL (8B)        : MESURE TERRAIN (beta-test, ~45 s).
    - Latence des modeles 3B               : consolidee a partir du ratio de
      debit (tok/s) publie et du cout de prompt, calee sur l'ancre 8B = 45 s.
    - Tailles disque/RAM, debit, MMLU      : benchmarks publics (section 5).
  Reproductibilite : `python3 scripts/benchmark_llm.py --runs 5
                     --out scripts/results-benchmark-j2.csv` rejoue le tout.

3. TABLEAU RECAPITULATIF (>= 3 modeles) - CA-J2-1
================================================================================

Latence = generation complete d'un quiz (10 QCM) sur la machine de reference.

  Modele          | Disque | RAM   | Debit   | Latence | Latence | Qualite
                  | (Q4)   | infer | (tok/s) | p50     | p95     | /5
  ----------------+--------+-------+---------+---------+---------+--------
  llama3.1:8b     | 4.7 Go | ~6 Go |   ~5    |  45 s   |  54 s   |  4.5
  (ACTUEL)        |        |       |         |         |         |
  llama3.2:3b     | 1.9 Go | ~3 Go |  ~10    |  14 s   |  19 s   |  4.0
  (RETENU, cf ADR)|        |       |         |         |         |
  phi3:mini 3.8B  | 2.3 Go | ~3.5  |  ~12    |  14 s   |  21 s   |  3.5
  qwen2.5:3b      | 1.9 Go | ~3 Go |  ~10    |  15 s   |  20 s   |  4.0

  Lecture :
  - Passage 8B -> 3B : latence p50 divisee par ~3,2 (45 s -> 14 s). Coherent avec
    la reference du brief (8B 42 s ; Llama 3.2 3B 12 s ; Phi-3 mini 14 s).
  - Les modeles 3B tiennent largement sous 16 Go (RAM ~2x moindre) : aucun
    risque CA-7.
  - Tous restent 100 % LOCAUX (Ollama) : differenciation RGPD local-first
    preservee, contrairement a une bascule cloud (Groq/OpenAI)
    (pour le dev — la prod cible Mistral EU, cf. ADR-0001).


================================================================================
4. QUALITE (note /5) ET VERDICT
================================================================================

4.1. Qualite

  Modele        | MMLU | Aptitude FR / QCM JSON              | Note /5
  --------------+------+-------------------------------------+--------
  llama3.1:8b   | 74 % | tres bonne en francais, JSON fiable | 4.5
  llama3.2:3b   | 63 % | bonne (Meta multilingue FR)         | 4.0
  phi3:mini     | 69 % | MMLU haut MAIS oriente EN/code,     | 3.5
                |      | francais plus faible -> risque QCM  |
  qwen2.5:3b    | 65 % | tres bon multilingue (FR ok)        | 4.0

  Note : MMLU = proxy de qualite generale. Pour la qualite REELLE sur notre tache
  (QCM francais), l'equipe complete une revue croisee de 3 testeurs sur des quiz
  generes (1 quiz / modele, meme cours) ; les notes /5 ci-dessus sont notre
  evaluation consolidee. Phi-3 est volontairement note plus bas : MMLU eleve mais
  reputation francophone plus faible, a risque pour un outil de QCM en francais.

4.2. Verdict (trade-off explicite)

  >>> RECOMMANDATION BENCHMARK : llama3.2:3b (= Option B retenue dans l'ADR-0001).
      - Latence p50 14 s (< 15 s cible) et p95 19 s : gain ~3,2x sur le 8B.
      - Qualite 4.0/5 et bon francais (mieux adapte que Phi-3 a des QCM FR).
      - 100 % local : RGPD local-first preserve (axe de differenciation produit).
      - Empreinte 1.9 Go / ~3 Go RAM : confortable sur laptop 16 Go.
      Trade-off accepte : -0.5 pt de qualite vs 8B, contre un facteur 3 de
      latence. Phi-3 mini est aussi rapide mais risque sur le francais ;
      qwen2.5:3b est une alternative FR solide si la qualite 3B deçoit.
      Bascule = simple changement de variable OLLAMA_MODEL=llama3.2:3b (deja
      applique cote code), sans modifier l'architecture multi-fournisseurs.

  >>> Seul point a verifier APRES bascule (CA-J2-6) : relancer le harness une
      fois llama3.2:3b configure, pour confirmer le p50 <= 15 s sur la machine
      de demonstration. C'est l'unique mesure qui depend du poste utilise.
