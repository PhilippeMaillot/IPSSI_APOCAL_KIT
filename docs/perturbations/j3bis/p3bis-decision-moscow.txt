1. CONTEXTE DE LA PERTURBATION
================================================================================

Le mercredi apres-midi, en pleine livraison MVP (Release 1 a 17h45), le PO
transmet une demande formelle de Hugo PETIT (M2 Droit numerique, Paris-Saclay),
utilisateur testeur du compte hugo.petit@test.local.

Demande recue (Art. 15 RGPD — droit d'acces) :
  - export integral des donnees personnelles detenues par EduTutor IA ;
  - format structure, couramment utilise et lisible par machine (JSON ou CSV) ;
  - categories attendues : compte, cours uploades, quiz generes, reponses,
    scores, signalements eventuels, logs d'audit ;
  - delai legal 30 jours, sponsor exige une reponse sous 48 h.

Cette perturbation ne demande pas un sprint de code immediat : elle impose une
DECISION de repriorisation (posture produit + conformite documentee). Elle
s'appuie sur la politique de retention deja redigee
(p3bis-politique-retention.txt) et sur les nouvelles user stories RGPD
(p3bis-product-backlog-maj.txt).

Articles RGPD mobilises :
  - Art. 13 : information des personnes (pages legales)
  - Art. 15 : droit d'acces (SAR — Subject Access Request)
  - Art. 17 : droit a l'effacement
  - Art. 20 : droit a la portabilite (format reutilisable)

3. DECISION PRODUIT — ARBITRAGE MVP vs CONFORMITE
================================================================================

Verdict equipe :
  La Release 1 (mercredi 17h45) est INVALIDE sans les elements RGPD MUST.
  Le MVP fonctionnel (F1-F6) reste livrable, mais la demonstration au PO
  doit inclure la posture conformite minimale.

Elements bloquants Release 1 (non negociables) :
  1. US-RGPD-01 / US-18 : GET /api/accounts/me/export/ + JSON + CSV
  2. US-RGPD-02 / US-05 : suppression compte + cascade donnees validee
  3. US-RGPD-03 : modele DataRequest + trace export/suppression
  4. US-RGPD-04 : 4 pages legales completees (plus de placeholders vides)
  5. Politique de retention publiee (reference p3bis-politique-retention.txt)
  6. US-RGPD-05 : reponse ecrite a Hugo Petit (p3bis-reponse-hugo.txt)

Elements deja en place dans le kit (gain de temps) :
  - Bouton suppression compte sur /profile (US-05 partiellement code)
  - 4 pages legales squelettes (/mentions-legales, /confidentialite, /cgu, /cookies)
  - Auth par token — filtrage request.user possible des la conception export

Elements a implementer en priorite Sprint 5 :
  - TASK-27 : endpoint export + bouton telechargement
  - TASK-28 : validation suppression cascade
  - TASK-26 : contenu pages legales + lien politique retention
  - Modele DataRequest + creation auto a chaque export/suppression


================================================================================
4. ALIGNEMENT AVEC LA POLITIQUE DE RETENTION
================================================================================

La repriorisation MoSCoW s'appuie sur p3bis-politique-retention.txt :

  Donnee              | Duree retenue        | Impact MoSCoW
  --------------------+----------------------+----------------------------------
  PDF originaux       | 30 jours max         | MUST documenter + SHOULD purger
  Cours / quiz /      | 12 mois              | MUST inclure dans export SAR
  scores / reponses   |                      |
  Logs applicatifs    | 30 jours             | MUST inclure si disponibles (Hugo)
  Audit trail SAR     | 3 ans (minimal)      | MUST — US-RGPD-03
  Compte actif        | Tant que compte      | MUST export + effacement

Decision : les durees de retention ne remettent pas en cause le MUST des
droits utilisateur ; elles contraignent le CONTENU de l'export et les pages
legales (US-RGPD-04).
