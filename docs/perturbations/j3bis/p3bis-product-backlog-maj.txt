================================================================================
0. CONTEXTE DE LA MISE A JOUR
=============================

La perturbation J3-bis introduit une contrainte RGPD forte : l'application doit
etre capable de repondre a une demande d'acces aux donnees personnelles et de
permettre a l'utilisateur d'exercer ses droits.

Le Product Backlog doit donc etre actualise avec de nouvelles user stories RGPD.

Objectifs :

* permettre a un utilisateur de recuperer ses donnees ;
* permettre a un utilisateur de supprimer ses donnees ;
* tracer les demandes RGPD dans un audit trail ;
* rendre visible la politique de retention et les mentions legales ;
* maintenir le MVP livrable sans faire exploser le perimetre.

Decision produit :
Les stories RGPD liees aux droits fondamentaux de l'utilisateur passent en MUST.
Les fonctionnalites plus avancees d'administration ou d'automatisation passent
en SHOULD ou COULD.

   ================================================================================

## US-RGPD-01 - Exporter ses donnees personnelles

Epic :
EPIC RGPD - Droits utilisateur

User story :
En tant qu'utilisateur authentifie,
je veux recuperer mes donnees personnelles au format JSON et CSV,
afin d'exercer mon droit d'acces et de portabilite.

Analyse INVEST :

* Independante : peut etre developpee via un endpoint dedie.
* Negociable   : le premier export peut contenir les donnees principales.
* Valuable     : repond directement au droit d'acces et de portabilite.
* Estimable    : les donnees exportees sont identifiables dans les modeles.
* Small        : premiere version limitee a l'utilisateur connecte.
* Testable     : on peut verifier le contenu du JSON et du CSV.

Priorite MoSCoW :
MUST

Justification MoSCoW :
Cette story est indispensable car elle repond a une exigence RGPD centrale.
L'utilisateur doit pouvoir recuperer ses donnees dans un format lisible par
machine. Elle doit etre integree au MVP ou documentee comme contrainte critique
a livrer immediatement.

Criteres d'acceptation :

* L'utilisateur authentifie peut acceder a un export de ses donnees.
* L'export contient uniquement ses propres donnees.
* L'export est disponible en JSON.
* L'export est disponible en CSV ou dans un format tabulaire equivalent.
* L'export ne contient aucun mot de passe, token ou secret technique.
* Une trace de la demande est creee dans l'audit trail SAR.

Impact DoR / DoD :
DoR :
* Lister les donnees exportables.
* Valider le format JSON et CSV attendu.
* Verifier les droits d'acces de l'utilisateur.

DoD :
* Endpoint GET /api/accounts/me/export/ fonctionnel.
* Export limite a l'utilisateur connecte.
* Tests effectues sur le contenu de l'export.
* Audit trail cree apres demande.

## US-RGPD-02 - Supprimer ses donnees personnelles

Epic :
EPIC RGPD - Droits utilisateur

User story :
En tant qu'utilisateur authentifie,
je veux supprimer mon compte et mes donnees personnelles,
afin d'exercer mon droit a l'effacement.

Analyse INVEST :

* Independante : peut etre developpee comme action compte separee.
* Negociable   : la premiere version peut supprimer ou anonymiser selon le cas.
* Valuable     : repond a un droit utilisateur fondamental.
* Estimable    : les donnees a supprimer sont liees au compte utilisateur.
* Small        : premiere version limitee aux donnees personnelles directes.
* Testable     : on peut verifier que les donnees ne sont plus accessibles.

Priorite MoSCoW :
MUST

Justification MoSCoW :
Cette story est prioritaire car elle repond au droit a l'effacement. Elle est
critique pour la conformite et pour la confiance utilisateur. Elle doit couvrir
au minimum le compte, les cours, les quiz, les reponses, les scores et
l'historique.

Criteres d'acceptation :

* L'utilisateur peut demander la suppression de ses donnees.
* Une confirmation explicite est demandee avant suppression.
* Le compte utilisateur est supprime ou desactive.
* Les cours, quiz, reponses, scores et historiques associes sont supprimes ou
  anonymises.
* Les donnees de l'utilisateur ne sont plus visibles dans les vues enseignantes.
* Une trace minimale de la demande est conservee dans l'audit trail SAR.

Impact DoR / DoD :
DoR :
* Definir les tables impactees par la suppression.
* Definir ce qui est supprime et ce qui est anonymise.
* Valider la conservation minimale de l'audit trail SAR.

DoD :
* Suppression testee sur un compte de test.
* Les donnees ne sont plus accessibles via l'API.
* Les donnees ne sont plus visibles cote frontend.
* Audit trail SAR cree sans recopier les donnees supprimees.

## US-RGPD-03 - Tracer les demandes RGPD

Epic :
EPIC RGPD - Audit trail SAR

User story :
En tant qu'administrateur,
je veux conserver une trace minimale des demandes RGPD,
afin de prouver que l'equipe a traite les demandes d'acces ou d'effacement.

Analyse INVEST :

* Independante : peut etre implementee avec un modele DataRequest.
* Negociable   : premiere version limitee a type, statut, date et utilisateur.
* Valuable     : permet de prouver le traitement d'une demande.
* Estimable    : necessite un modele simple et quelques statuts.
* Small        : pas besoin d'un back-office complet pour le MVP.
* Testable     : on peut verifier qu'une demande cree une entree en base.

Priorite MoSCoW :
MUST

Justification MoSCoW :
Cette story est liee a la capacite de repondre proprement aux demandes SAR.
Elle est en MUST car l'equipe doit pouvoir demontrer que la demande a ete prise
en compte, sans stocker inutilement toutes les donnees personnelles.

Criteres d'acceptation :

* Un modele DataRequest existe.
* Une demande d'export cree une entree d'audit.
* Une demande de suppression cree une entree d'audit.
* L'entree contient au minimum : utilisateur, type de demande, date, statut.
* L'audit trail ne contient pas les donnees personnelles exportees en clair.

Impact DoR / DoD :
DoR :
* Definir les statuts possibles : pending, completed, failed.
* Definir les types de demande : export, delete.

DoD :
* DataRequest cree en base.
* Creation automatique d'une trace lors d'un export ou d'une suppression.
* Verification que les donnees sensibles ne sont pas dupliquees inutilement.

## US-RGPD-04 - Consulter les mentions legales et la politique de retention

Epic :
EPIC RGPD - Transparence

User story :
En tant qu'utilisateur,
je veux consulter les mentions legales et la politique de retention des donnees,
afin de comprendre comment EduTutor IA traite mes donnees personnelles.

Analyse INVEST :

* Independante : peut etre developpee comme page statique.
* Negociable   : contenu minimal acceptable pour le MVP.
* Valuable     : renforce la transparence et la confiance.
* Estimable    : faible complexite technique.
* Small        : une page simple suffit pour la premiere version.
* Testable     : on peut verifier que la page est accessible.

Priorite MoSCoW :
SHOULD

Justification MoSCoW :
Cette story est importante pour la transparence RGPD, mais elle peut etre livree
sous forme simple. Elle accompagne les stories MUST d'export, de suppression et
d'audit trail.

Criteres d'acceptation :

* Une page mentions legales est accessible depuis l'application.
* La politique de retention est visible ou referencee.
* Les droits utilisateur sont expliques simplement.
* Les formats d'export JSON / CSV sont mentionnes.
* Le contact ou canal de demande RGPD est indique si disponible.

Impact DoR / DoD :
DoR :
* Valider le contenu minimal avec le PO.
* Verifier les liens vers les pages legales.

DoD :
* Page accessible depuis le frontend.
* Contenu relu par l'equipe.
* Aucun texte placeholder vide ne reste en production.

================================================================================
2. PRIORISATION MOSCOW ACTUALISEE
=================================

## MUST - Indispensable

* US-RGPD-01 : exporter ses donnees personnelles en JSON / CSV.
* US-RGPD-02 : supprimer ses donnees personnelles.
* US-RGPD-03 : tracer les demandes RGPD avec un audit trail SAR.
* Maintenir le parcours MVP : compte, upload, generation, correction, score,
  historique.

Justification :
Ces stories repondent aux droits fondamentaux de l'utilisateur. Elles sont
critiques pour eviter un MVP fonctionnel mais non conforme.

## SHOULD - Important mais negociable

* US-RGPD-04 : consulter les mentions legales et la politique de retention.
* Ajouter un bouton export visible dans le frontend.
* Ajouter une confirmation explicite avant suppression.
* Documenter les limites de retention dans le depot.

Justification :
Ces elements renforcent la transparence et l'utilisabilite, mais une premiere
version simple peut suffire pour le MVP.

## COULD - Interessant mais reportable

* Interface admin complete de suivi des demandes RGPD.
* Purge automatique planifiee.
* Export avance par periode ou par type de donnee.
* Historique detaille des exports telecharges.

Justification :
Ces elements apportent de la maturite mais ne sont pas indispensables pour
repondre a la perturbation dans le temps disponible.

## WON'T HAVE - Hors perimetre immediat

* Gestion juridique complete multi-etablissements.
* Workflow avance de validation DPO.
* Connecteurs externes pour exporter vers d'autres plateformes.
* Anonymisation statistique avancee multi-classes.

Justification :
Ces fonctionnalites depassent le perimetre du MVP et risquent de faire exploser
le scope de la livraison.

================================================================================
3. IMPACT SUR LE PRODUCT BACKLOG EXISTANT
=========================================

Stories existantes impactees :

* Inscription / connexion : verifier que l'utilisateur authentifie est bien le
  proprietaire des donnees exportees.
* Profil utilisateur : ajouter un acces a l'export ou a la suppression.
* Historique des quiz : rendre les donnees exportables.
* Score et detail des corrections : inclure dans l'export.
* Suivi enseignant : retirer ou anonymiser les donnees d'un etudiant supprime.

Nouvelles dependances :

* Endpoint REST GET /api/accounts/me/export/
* Modele DataRequest pour audit trail SAR
* Bouton export frontend
* Action ou demande de suppression des donnees
* Page mentions legales / politique de retention

================================================================================
4. CRITERES D'ACCEPTATION GLOBAUX
=================================

Le Product Backlog RGPD actualise est accepte si :

* au moins une user story couvre l'export des donnees ;
* au moins une user story couvre la suppression des donnees ;
* les formats JSON et CSV sont mentionnes ;
* les criteres d'acceptation sont testables ;
* les priorites MoSCoW sont visibles et justifiees ;
* les impacts DoR / DoD sont explicites ;
* les stories ne cassent pas le MVP principal ;
* les taches techniques correspondantes peuvent etre reprises dans le Sprint
  Backlog.

================================================================================
5. SYNTHESE
===========

La perturbation J3-bis ajoute une exigence de conformite au Product Backlog.

Le MVP ne peut pas seulement etre fonctionnel : il doit aussi permettre a
l'utilisateur d'exercer ses droits sur ses donnees.

Les stories d'export, de suppression et d'audit trail passent en MUST, car elles
sont directement liees a la demande RGPD. Les pages legales et la politique de
retention sont importantes et classees en SHOULD si elles sont livrees sous forme
simple. Les fonctions avancees d'administration et d'automatisation sont repoussees
en COULD ou WON'T HAVE.

Cette mise a jour permet de traiter la perturbation sans remettre en cause le
MVP, mais en ajoutant les exigences minimales de conformite et de transparence.
