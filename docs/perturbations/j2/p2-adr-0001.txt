ADR-0001 : Choix du modele LLM pour la generation de QCM
==========================================================


1. CONTEXTE
===========
La beta-testeuse signale une latence de 45 s pour la generation de 10 QCM
avec llama3.1:8b sur CPU (rapport APOCAL'IPSSI J2, 30/06/2026).
L'objectif fixe par le PO est d'atteindre <= 15 s sur le cours de reference.

Le projet supporte plusieurs backends LLM via la variable LLM_BACKEND
(cf. .env.example). L'environnement de dev tourne sur CPU local sans GPU.
L'environnement de production cible un VPS OVH sans GPU
(cf. .env.prod.example : "Recommande sur VPS sans GPU : un fournisseur CLOUD").


2. OPTIONS ENVISAGEES
=====================

Option A : Garder llama3.1:8b (Ollama local)
  p50=45 s, p95=54 s. Ne repond pas a l'objectif. Rejete.

Option B : llama3.2:3b (Ollama local)
  Estime p50=14 s sur machine ref (16 Go RAM, 8 coeurs).
  MESURE terrain machine equipe : p50 ~62 s -> objectif <= 15 s non tenu sur CPU standard.
  Rejete (revise en v1.1 apres mesure terrain — cf. p2-benchmark.txt section 3b).

Option C : phi3:mini 3.8B (Ollama local)
  p50=14 s, p95=21 s, qualite 3.5/5. Qualite insuffisante en francais. Rejete.

Option D : qwen2.5:3b (Ollama local)
  p50=15 s, p95=20 s, qualite 4/5. p50 a la limite du seuil. Moins marge. Rejete.

Option E : Mistral small (cloud EU)      <-- RETENU DEV + PROD (v1.1)
  Latence ~2-3 s, qualite elevee, donnees en Europe (RGPD), free tier.
  Deja prevu dans .env.prod.example et supporte dans le codebase (mistral_client.py).
  Cle API gratuite : console.mistral.ai (inscription en 2 min).


3. DECISION (v1.1 — apres mesure terrain)
==========================================
DEV + PROD : mistral-small-latest via API Mistral AI (fournisseur europeen)

Motif de revision : mesure terrain llama3.2:3b = ~62 s sur machine equipe
(vs 14 s estime sur machine ref). Objectif <= 15 s intenable en Ollama local CPU.

Modification requise :
  .env (dev)       -> LLM_BACKEND=mistral
                      MISTRAL_MODEL=mistral-small-latest
                      MISTRAL_API_KEY=<cle console.mistral.ai, free tier>
  .env.prod (prod) -> idem (deja prevu dans .env.prod.example)


4. JUSTIFICATION (v1.1)
=======================
Revision post-mesure terrain :
  - llama3.2:3b estime 14 s sur machine ref, mesure ~62 s sur machine equipe
  - Ecart explique par la puissance CPU : l'inference Ollama est lineaire avec
    les coeurs et la frequence. Le projet lui-meme note "Sur CPU, un 3B met
    souvent 1 a 3 min" (.env commentaire). La valeur 14 s = materiel haut gamme.
  - Objectif <= 15 s intenable en Ollama local sur notre materiel

Mistral EU retenu pour dev ET prod :
  - Latence ~2-3 s : objectif <= 15 s tres largement atteint
  - Fournisseur europeen (France) : donnees en UE, conforme RGPD
  - Free tier : zero cout supplementaire pour le dev et la demo
  - Coherence dev/prod : meme backend, meme comportement, moins de divergences
  - Deja supporte dans le codebase (LLM_BACKEND=mistral, mistral_client.py)
  - Aucune modification de code : uniquement variables d'environnement


5. CONSEQUENCES (v1.1)
======================
Positives :
  + Objectif <= 15 s largement atteint (~2-3 s, facteur ~20x sur le modele initial)
  + RGPD preserve : Mistral AI, fournisseur francais, donnees en UE
  + Free tier : pas de cout pour le dev et la demo de la semaine
  + Coherence dev/prod : meme backend, moins de divergences de comportement
  + Plan de migration deja prevu et documente dans le projet

Negatives / Risques :
  - Connexion internet requise en dev (plus 100% local)
  - Cle API a generer avant la demo (console.mistral.ai, 2 min, gratuit)
  - Dependance externe : si API Mistral indisponible -> LLM_BACKEND=mock
  - llama3.2:3b reste disponible en Ollama pour dev offline si besoin
