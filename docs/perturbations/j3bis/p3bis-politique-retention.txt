0. OBJECTIF DU DOCUMENT
=======================

Cette politique de retention definit quelles donnees EduTutor IA conserve,
pendant combien de temps, pour quelle finalite, et comment elles sont supprimees
ou exportees lorsqu'un utilisateur exerce ses droits RGPD.

Elle repond a la perturbation J3-bis : l'equipe doit etre capable de traiter une
demande d'acces aux donnees personnelles, de fournir un export lisible par
machine, et de clarifier la retention des donnees.

Principes retenus :

* Minimisation : ne conserver que les donnees utiles au service.
* Transparence : expliquer clairement ce qui est stocke.
* Limitation de conservation : ne pas garder les donnees indefiniment.
* Portabilite : permettre un export JSON et CSV.
* Effacement : permettre la suppression des donnees utilisateur.
* Audit trail : tracer les demandes RGPD sans exposer inutilement les donnees.

================================================================================

1. PERIMETRE DES DONNEES CONSERVEES
   ================================================================================

EduTutor IA traite les categories de donnees suivantes :

1. Donnees de compte

* email ;
* identifiant utilisateur ;
* mot de passe chiffre / hash ;
* date de creation du compte ;
* date de derniere connexion.

2. Donnees de profil

* nom ou pseudonyme si renseigne ;
* role utilisateur : etudiant, enseignante, administrateur ;
* preferences simples si disponibles.

3. Donnees pedagogiques

* cours saisis en texte ;
* fichiers PDF uploades ou texte extrait du PDF ;
* quiz generes ;
* questions et propositions de reponse ;
* bonnes reponses ;
* reponses donnees par l'utilisateur ;
* score obtenu ;
* historique des quiz.

4. Donnees de suivi enseignant

* rattachement etudiant / classe / enseignante si mis en place ;
* scores consultables par l'enseignante ;
* indicateur simple de decrochage si developpe.

5. Donnees techniques

* logs applicatifs utiles au diagnostic ;
* erreurs techniques ;
* horodatage des actions importantes.

6. Donnees RGPD / SAR

* date de demande d'export ;
* date de demande de suppression ;
* statut de la demande ;
* identifiant de l'utilisateur concerne ;
* trace d'execution de la demande.

3. REGLES DE RETENTION DETAILLEES
=================================

3.1. Donnees de compte

Les donnees de compte sont conservees tant que l'utilisateur dispose d'un compte
actif. Elles sont necessaires au fonctionnement du service.

En cas de suppression du compte :

* le compte est supprime ;
* l'email est supprime ou anonymise ;
* les sessions actives sont invalidees ;
* les donnees pedagogiques associees sont supprimees ou anonymisees selon le cas.

3.2. PDF uploades

Les fichiers PDF uploades ne doivent pas etre conserves indefiniment.

Regle retenue :

* le PDF original est conserve 30 jours maximum ;
* si le texte extrait suffit au fonctionnement du service, le PDF peut etre
  supprime plus tot ;
* l'utilisateur peut demander la suppression immediate de ses fichiers.

Justification :
Le PDF peut contenir des donnees personnelles ou des informations sensibles.
Sa conservation doit donc etre limitee.

3.3. Cours, quiz, reponses et scores

Les cours, quiz, reponses et scores sont utiles pour l'historique utilisateur et
le suivi de progression.

Regle retenue :

* conservation 12 mois apres creation ;
* suppression automatique ou anonymisation apres expiration ;
* suppression immediate possible sur demande utilisateur.

Justification :
Cette duree couvre un cycle raisonnable de revision et d'annee scolaire sans
conserver les donnees inutilement.

3.4. Donnees de suivi enseignant

Les donnees liees a une classe ou a une enseignante sont conservees uniquement
pour l'annee scolaire ou la periode d'utilisation active.

Regle retenue :

* conservation pendant l'annee scolaire en cours ;
* suppression ou anonymisation en fin de periode ;
* l'enseignante ne voit que les donnees des etudiants rattaches a sa classe.

Point d'attention :
Si un etudiant exerce son droit a l'effacement, ses donnees personnelles ne
doivent plus apparaitre dans les vues enseignantes.

3.5. Logs applicatifs

Les logs techniques sont conserves pour diagnostiquer les erreurs et securiser
l'application.

Regle retenue :

* conservation 30 jours ;
* pas de conservation de contenu complet de cours dans les logs ;
* pas de stockage de mot de passe, token ou donnee sensible en clair.

3.6. Audit trail SAR

Les demandes RGPD doivent etre tracees pour prouver que l'equipe a repondu a la
demande.

Regle retenue :

* conservation 3 ans ;
* stockage minimal : type de demande, date, statut, utilisateur concerne ;
* ne pas recopier toutes les donnees personnelles dans l'audit trail.

================================================================================
4. EXPORT DES DONNEES
=====================

L'utilisateur doit pouvoir recuperer ses donnees dans un format lisible par
machine.

Formats retenus :

* JSON : export complet structure.
* CSV : export simplifie des donnees principales.

Donnees exportables :

* informations du compte ;
* profil ;
* quiz crees ou passes ;
* scores ;
* reponses ;
* historique ;
* demandes RGPD associees.

Endpoint cible :
GET /api/accounts/me/export/

Critere de conformite :
L'export doit etre comprehensible, reutilisable et limite aux donnees de
l'utilisateur authentifie.

================================================================================
5. SUPPRESSION DES DONNEES
==========================

L'utilisateur doit pouvoir demander la suppression de ses donnees.

Regles :

* supprimer ou anonymiser le compte ;
* supprimer les cours, PDF, quiz, reponses et scores associes ;
* supprimer les rattachements classe si l'utilisateur est etudiant ;
* conserver uniquement une trace minimale de la demande dans l'audit trail SAR ;
* ne pas supprimer les donnees necessaires a la preuve de traitement RGPD, mais
  les limiter au strict minimum.

Point d'attention :
Si certaines donnees sont visibles par une enseignante, la suppression d'un
etudiant doit retirer ses donnees personnelles des vues enseignantes.

================================================================================
6. SECURITE ET ACCES AUX DONNEES
================================

Principes :

* un utilisateur ne peut exporter que ses propres donnees ;
* une enseignante ne peut consulter que les donnees des etudiants rattaches a
  sa classe ;
* un administrateur ne doit pas pouvoir lire inutilement les contenus de cours ;
* les exports doivent etre generes a la demande, pas stockes publiquement ;
* les exports ne doivent jamais contenir de mot de passe, token ou secret.

Mesures minimales :

* authentification obligatoire ;
* verification du proprietaire des donnees ;
* journalisation de la demande ;
* limitation des donnees exportees au strict necessaire.

================================================================================
7. IMPACT SUR LE MVP
====================

La conformite RGPD devient une exigence non-fonctionnelle critique.

Ce qui doit etre integre rapidement :

* page mentions legales ;
* politique de retention ;
* endpoint d'export utilisateur ;
* bouton export dans l'interface ;
* suppression des donnees ou droit a l'effacement ;
* modele DataRequest pour l'audit trail SAR.

Ce qui peut etre differe :

* interface avancee d'administration des demandes RGPD ;
* automatisation complete de purge planifiee ;
* exports avances par classe ou etablissement ;
* tableau de bord juridique complet.

================================================================================
8. SYNTHESE DE LA DECISION
==========================

L'equipe retient une politique de retention simple et defendable :

* PDF originaux : 30 jours maximum.
* Logs techniques : 30 jours.
* Cours, quiz, reponses, scores : 12 mois.
* Donnees de classe : annee scolaire en cours.
* Audit trail SAR : 3 ans avec donnees minimales.

Cette politique permet de respecter les principes de minimisation, limitation de
conservation, portabilite et effacement, tout en conservant un MVP realiste pour
la livraison.
